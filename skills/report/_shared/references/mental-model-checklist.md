# 用户心智构建核查清单

这份 checklist 用于判断一份报告是否让读者形成了完整、正确、可复用的项目心智。它被 `report-audit` 列为必查维度之一；当 report 类型是 `executor-handbook` 时，它是硬性验收项。

面向的读者画像是「认真、愿意学，但不预设该领域背景」的中文 rookie 读者。判断标准不是读者是否承认「读懂了」，而是读者是否具备显式的、可检验的表达与推理能力。

## 核心验收问题

下面 8 个问题是 audit 的最小集合。每一条都应能对应到报告里的具体章节、图或段落。

### 1. 系统总图复现能力

- 触发条件：报告覆盖多模块或多层架构。
- 自检提示：读者合上报告后，是否还能手画出系统总图，并标出主要模块与数据流向。
- 修复建议：若无法复现，检查 `01-mental-model-overview` 或 `04-system-architecture` 是否提供了总览图与分层图，且图前引导与图后结论齐全。

### 2. 关键模块列举能力

- 触发条件：报告存在 3 个以上职责不同的模块。
- 自检提示：读者是否能列出 3--5 个关键模块及其职责、输入输出、主要依赖。
- 修复建议：若列不出，检查模块职责是否写进了独立段落或表格，而不是被融在长段背景里。

### 3. 核心 tradeoff 复述能力

- 触发条件：存在至少一个被显式选择的技术路线。
- 自检提示：读者是否能说出每个关键设计决策的主要 tradeoff，以及替代方案被排除的理由。
- 修复建议：若复述不出，检查 `05-core-module-design` 中各模块的设计决策表是否包含 决策 | 选择 | 备选 | 理由 四列。

### 4. 概念首次定义定位能力

- 触发条件：报告引入了至少 5 个领域术语。
- 自检提示：读者是否能快速定位任一关键概念的首次定义位置。
- 修复建议：若定位困难，检查术语表章节是否完整，且正文首次出现时是否使用 `\textbf{}` 或 `\term{}` 显式标记。

### 5. 数据路径贯通能力

- 触发条件：报告描述输入 → 处理 → 输出型系统。
- 自检提示：读者是否能说出数据从入口到出口的完整路径，以及每一跳的关键变换与失败可能。
- 修复建议：若跳跃感强，检查 `06-core-workflow` 是否给出时序图且按真实顺序标注关键步骤。

### 6. 证据层区分能力

- 触发条件：报告同时包含外部资料、设计意图与当前实现状态。
- 自检提示：读者是否能对每个重要判断归类为 `source claim` / `design intent` / `repo-observed fact` / `report synthesis`。
- 修复建议：若混淆，检查现状事实是否被错误地写进了非 `项目进度` 章节。

### 7. 主问题回答能力

- 触发条件：报告有明确的执行摘要或研究问题。
- 自检提示：读者是否能用两三句话回答执行摘要提出的主问题。
- 修复建议：若回答跑题，检查全文结论与执行摘要是否一致，是否存在章节回答了不同的问题。

### 8. 替代方案排除能力

- 触发条件：关键决策存在合理替代。
- 自检提示：读者是否能解释为什么不采用 N 个替代方案。
- 修复建议：若只看到「我们选了 A」没看到「为什么不是 B/C」，进入 `report-audit` 多智能体审计，补强反方压力和证据边界。

## 每章小结的强约束

每一章结尾必须有一句明确的「本节读完后，读者应能……」。这一句不是装饰，而是 audit 的原子。当某章找不到这句小结，或小结与章内内容偏离时，立刻视为 mental-model 缺口。

推荐写法：

- `本节读完后，读者应能画出系统总图并标出四个主要模块及其交互方向。`
- `本节读完后，读者应能解释为什么选择 A 方案，以及 B 方案的主要代价是什么。`

禁止写法：

- `本节介绍了……`（只告诉读者章节讲了什么，没告诉读者应形成什么能力）
- `本节小结。`（零信息）

## 与其他验收的关系

- 与 `executor-report-style.md`：风格层负责句式，本文件负责读者能力。二者互补。
- 与 `implementation-readiness-checklist.md`：本文件关心「读者是否建立心智」，后者关心「心智是否足以指导实现」。
- 与 `rookie-simulator-protocol.md`：本文件是静态 checklist，后者是动态走查。audit 流程中先静态检查本文件的 8 条，若 3 条以上失败，再启动 rookie walkthrough。

## 自检执行顺序

建议按以下顺序扫一遍：

1. 扫每章是否都有「本节读完后，读者应能……」小结句。
2. 扫执行摘要与末章是否回答同一个主问题。
3. 扫术语表完整性与首次定义位置一致性。
4. 扫模块图、数据流图、tradeoff 表是否都存在且被引用。
5. 剩余 4 条（6、8 及其余）按报告内容触发。

缺口出现频率高的章节，通常是 `01-mental-model-overview`、`04-system-architecture`、`05-core-module-design`。
