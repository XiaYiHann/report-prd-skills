# 完整项目报告模板（金字塔 + 图解版 + PRD 融合）

在用户明确要"给我一个直接可用的报告模板""按模板搭一个完整项目报告""给学生/执行层也能看懂的项目手册"或"工程 PRD"时，优先参考这份模板。

**核心变化**：从"文字驱动的章节列表"变为"图解驱动的金字塔结构"，同时融入 Vibe Coding / Agentic 时代的 PRD 最佳实践（Goals & Non-Goals、模块化 Acceptance Criteria、Phased MVP Roadmap）。

每章的结构是：结论 → MECE 支撑 → 图解展开 → 细节补充。

---

## 设计原则

- **结构原则**：结论先行，以上统下
- **图解原则**：能用图说的不用字，每章标配至少一张图
- **MECE 原则**：支撑要点相互独立、完全穷尽
- **叙事原则**：先结论，再支撑，后细节
- **教学原则**：先讲为什么，再讲是什么和怎么做
- **执行原则**：每个模块都要能被单独展开为「架构图 + 接口表 + 时序图」
- **现状原则**：当前实现、里程碑与阻塞默认集中写入 `项目进度`
- **PRD 原则**：Goals & Non-Goals 明确分离，功能需求模块化，每个模块带 Acceptance Criteria
- **AI 友好**：清晰列表、具体引用、可测试标准，便于喂给 AI agent

## 图表模板

### 默认图表策略

| 内容类型 | 默认图型 | 默认表型 | 主要作用 |
| --- | --- | --- | --- |
| 架构 / 框架 | 总览图 + 分层图 + 模块图 | 模块职责表 | 让读者先建立全局模型 |
| 流程 / 算法 | 时序图或流程图 | 步骤-输入-输出表 | 让读者看懂执行顺序 |
| 模块设计 | 模块架构图 | 组件职责表 | 让读者看懂模块边界 |
| 接口契约 | Mermaid source 数据流图或 TikZ 数据流图 | 字段定义表 | 让读者对齐输入输出 |
| 实验 / 证据 | 结果图 | baseline / ablation / evidence ledger 表 | 让读者看懂结论和证据边界 |
| 风险 / 局限 | 边界图 | claim-evidence-risk / failure-case 表 | 让读者看懂哪里不能过度外推 |
| 用户旅程 | 旅程地图 | 步骤-情绪曲线表 | 让读者理解用户体验 |

### 图前图后写法

- **图前**：说明这张图回答什么问题，读者应该关注什么
- **图后**：明确一句"读者应该注意什么"的结论
- **图注**：优先写结论，而不是只写对象名

**示例**：
```markdown
信号从生成到订单落地的完整链路如图 X 所示。读者应关注三个决策点：
风控评估、账户验证、交易所响应超时处理。

[图 X：时序图]

图 X 显示，风控决策是执行主路径中的阻塞点：任何信号必须通过风控评估
后才能进入下单环节。这一设计确保风控不可绕过。
```

## 可选分离（Warp 风格）

对于大型项目，可将报告分为两个文件：

- **PRODUCT.md**：用户行为、Success Criteria、Edge Cases、User Journeys、Goals & Non-Goals
- **TECH.md**：Relevant Code、Proposed Changes、Testing Plan、Architecture Decisions、Agent Rules

分离原则：
- 产品经理/设计师主要维护 PRODUCT.md
- 工程师/架构师主要维护 TECH.md
- 两个文件通过交叉引用保持同步

## 章节质量检查表

每章必须满足：

- [ ] 开头有明确结论句
- [ ] 支撑结论的要点 MECE
- [ ] 有至少一张图或表
- [ ] 模块章节有架构图 + 组件表
- [ ] 流程章节有时序图 + 步骤表
- [ ] 图前有引导（说明图回答什么问题）
- [ ] 图后有结论（说明读者应带走什么判断）
- [ ] 无对照否定句（不是...而是...）
- [ ] 无纯描述性段落
- [ ] 无与其他章节重复的内容

**PRD 特殊检查：**

- [ ] Goals 和 Non-Goals 已明确分离
- [ ] 每个功能模块都有 Acceptance Criteria（可测试清单）
- [ ] 每个功能都有优先级（P0/P1/P2）和依赖声明
- [ ] 非功能需求有具体量化指标
- [ ] Tech Stack 有版本号和选择理由
- [ ] Agent Rules / Prompt 指南已定义（若使用 AI 辅助开发）
- [ ] MVP Roadmap 有明确的验收标准和 Go/No-Go 决策点
- [ ] 用户画像和使用场景已覆盖核心路径和异常分支

## AI 协作拆章建议

把报告拆成可独立协作的模块，每章都尽量满足下面四项：

- **定义**：本章要回答什么
- **规范**：本章必须覆盖哪些点（含图/表要求）
- **图解**：需要几张图，每张图回答什么问题
- **验收**：读完后读者应该理解什么、能做什么

对于 vibe-coding 工作流，推荐按以下顺序喂给 AI：

1. **先喂**：Goals & Non-Goals + 用户画像（建立上下文）
2. **再喂**：功能需求模块（带 Acceptance Criteria，逐模块生成）
3. **最后喂**：技术与实现指南（让 AI 在已有代码基础上做修改）
