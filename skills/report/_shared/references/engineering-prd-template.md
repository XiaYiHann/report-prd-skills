# 工程项目 PRD 模板（Vibe Coding / Agentic 时代）

在用户明确要"工程 PRD""产品需求文档""设计文档""vibe-coding 项目文档"或 AI-agent 驱动的项目时，优先参考这份模板。

**核心理念**：AI 可执行 + 人类 oversight + 快速迭代。核心 5-15 页，高度模块化。

**最佳实践**：
- 模块化 + Acceptance Criteria 驱动 AI
- Living Document + 版本控制
- 配套 AGENTS.md / Rules 文件
- 先 PRD → AI 生成 → 人类 review 迭代

---

## 设计原则

- **AI 友好**：清晰列表、具体引用、可测试标准
- **模块化**：按组件/页面/服务拆分，每模块可独立交付
- **验收驱动**：每个功能点都有可测试的 Acceptance Criteria
- **迭代优先**：Phased MVP，短周期、快反馈
- **人机协作**：PRD 给 AI 提供上下文，人类负责 review 和决策

- [ ] 用户可以通过邮箱+密码登录

## 可选分离（Warp 风格）

对于大型项目，可将 PRD 分为两个文件：

- **PRODUCT.md**：用户行为、Success Criteria、Edge Cases、User Journeys
- **TECH.md**：Relevant Code、Proposed Changes、Testing Plan、Architecture Decisions

分离原则：
- 产品经理/设计师主要维护 PRODUCT.md
- 工程师/架构师主要维护 TECH.md
- 两个文件通过交叉引用保持同步

## 章节质量检查表

每章必须满足：

- [ ] 开头有明确结论句
- [ ] 支撑结论的要点 MECE
- [ ] 有至少一张图或表
- [ ] 模块章节有架构图 + 组件表
- [ ] 流程章节有时序图 + 步骤表
- [ ] 图前有引导（说明图回答什么问题）
- [ ] 图后有结论（说明读者应带走什么判断）
- [ ] 无对照否定句（不是...而是...）
- [ ] 无纯描述性段落
- [ ] 无与其他章节重复的内容

**工程 PRD 特殊检查：**

- [ ] Goals 和 Non-Goals 已明确分离
- [ ] 每个功能模块都有 Acceptance Criteria（可测试清单）
- [ ] 每个功能都有优先级（P0/P1/P2）和依赖声明
- [ ] 非功能需求有具体量化指标
- [ ] Tech Stack 有版本号和选择理由
- [ ] Agent Rules / Prompt 指南已定义（若使用 AI 辅助开发）
- [ ] MVP Roadmap 有明确的验收标准
- [ ] API 规范包含输入/输出/错误码
- [ ] 环境搭建命令可复制执行

## AI 协作拆章建议

把报告拆成可独立协作的模块，每章都尽量满足下面四项：

- **定义**：本章要回答什么
- **规范**：本章必须覆盖哪些点（含图/表要求）
- **图解**：需要几张图，每张图回答什么问题
- **验收**：读完后读者应该理解什么、能做什么

对于 vibe-coding 工作流，推荐按以下顺序喂给 AI：

1. **先喂**：Goals & Non-Goals + 用户画像（建立上下文）
2. **再喂**：功能需求模块（带 Acceptance Criteria，逐模块生成）
3. **最后喂**：技术与实现指南（让 AI 在已有代码基础上做修改）
