# 实现就绪度核查清单

这份 checklist 用于判断「读者能否仅凭这份报告完成一次可验证的实现」。当报告类型是 `executor-handbook` 时它是硬性验收项；其他类型可作为可选增强。

它不要求报告给出完整代码，但要求每一个执行路径可被读者复现，每一个接口与 DoD 显式可判定。

## 7 类验收项

### 1. 前置工具与版本

- 必须章节：`08-environment-and-setup`。
- 最小结构：工具名、版本范围、用途、获取方式各占一列的表格。
- 失败信号：
  - 仅写「需要 Python 环境」不写版本；
  - 工具名散落在正文，没有集中表格；
  - 版本被写成「较新版本」「最新稳定版」。

### 2. 访问凭证与权限

- 必须章节：`08-environment-and-setup`。
- 最小结构：凭证名、作用域、申请路径、有效期、是否可在本地缓存。
- 失败信号：
  - 提到需要 API key 但不说从哪申请；
  - 混用 token 类型却没有区分；
  - 没有说明凭证丢失时的重置流程。

### 3. 环境搭建命令可复制

- 必须章节：`08-environment-and-setup`。
- 最小结构：连续、可从上到下复制执行的命令块；每条命令前有一行说明其产物或副作用。
- 失败信号：
  - 命令块里混入占位符却未标记（例如裸的 `<your-token>`）；
  - 命令依赖未声明的环境变量；
  - 命令顺序与实际执行顺序不符。

### 4. 接口契约完整

- 必须章节：`07-interfaces-and-contracts`。
- 最小结构：对每个外部暴露接口给出：请求方式、路径或函数签名、输入字段与类型、输出字段与类型、错误码、幂等性与副作用。
- 失败信号：
  - 仅有接口名没有字段；
  - 错误码未覆盖或写成「出错时返回错误」；
  - 副作用未声明（例如是否写库、是否发消息）。

### 5. 数据模型显式

- 必须章节：`07-interfaces-and-contracts` 或同级数据章节。
- 最小结构：对每个核心数据对象给出：字段名、类型、含义、约束、示例值。
- 失败信号：
  - 仅用散文描述字段；
  - 字段间关系只在图里出现，没有文字说明；
  - 约束（非空、枚举值、长度）缺失。

### 6. 最小 Hello World 可跑

- 必须章节：`09-implementation-walkthrough`。
- 最小结构：目标、输入、操作步骤、预期输出、验证方式各一段；操作步骤里的命令应可从上到下复制执行。
- 失败信号：
  - 走查描述跳过真实命令；
  - 预期输出被写成「类似如下」却未给出可对比的样例；
  - 没有说明若输出不符应如何定位。

### 7. DoD 显式

- 必须章节：`10-testing-and-acceptance`。
- 最小结构：每个交付单元的 Definition of Done 写成可判定句（是/否）。
- 失败信号：
  - DoD 写成「功能可用」「质量达标」；
  - 缺失边界情况的验收；
  - 只列了功能 DoD，没有文档或演示 DoD。

## 章节映射表（针对 executor-handbook）

| 验收项 | 对应章节 | 必须出现位置 |
| --- | --- | --- |
| 前置工具与版本 | `08-environment-and-setup` | 章节开头的工具表 |
| 访问凭证与权限 | `08-environment-and-setup` | 紧随工具表的凭证表 |
| 环境搭建命令可复制 | `08-environment-and-setup` | 章节主体命令块 |
| 接口契约完整 | `07-interfaces-and-contracts` | 按接口顺序的契约表 |
| 数据模型显式 | `07-interfaces-and-contracts` | 紧随接口契约的数据对象表 |
| 最小 Hello World 可跑 | `09-implementation-walkthrough` | 章节主体 |
| DoD 显式 | `10-testing-and-acceptance` | 章节末尾的 DoD 表 |

其他章节（`01-mental-model-overview`、`04-system-architecture` 等）不承担实现就绪度验收，但可以引用本清单中的表格。

## 自检执行顺序

1. 先检查 `08-environment-and-setup` 是否三段完整：工具表、凭证表、命令块。
2. 再检查 `07-interfaces-and-contracts` 是否存在接口契约表与数据对象表。
3. 再执行 `09-implementation-walkthrough` 的最小 Hello World，判断命令能否真实跑通（必要时交给 `rookie-simulator-protocol.md` 的走查步骤）。
4. 最后检查 `10-testing-and-acceptance` 的 DoD 是否每条都可判定。

任一步失败就视为实现就绪度不达标，不应把对应报告当成下游执行依据。

## 与其他验收的关系

- 本清单判断「报告能否支持动手实现」，`mental-model-checklist.md` 判断「读者是否建立了心智」。心智不足时，就绪度也站不住。
- 本清单与 `progress-chapter-policy.md` 互补：前者要求执行路径可复现，后者要求仓库现状集中在 `项目进度`。避免把凭证、命令等实现细节错误写进 `项目进度`。
