# Rookie Walkthrough 协议

这份 reference 定义 `report-audit` 中「假装一位零背景读者逐节走查」的步骤。它是 audit 的可选步骤，但一旦触发，应按本协议执行，不得即兴。

## 目的

静态 checklist 能发现结构性缺口，但不能发现「叙事顺序把读者带偏了」这类问题。rookie walkthrough 用一位合格但无领域背景的模拟读者，按推荐阅读路径走一遍报告，把每一次卡顿都记录下来。输出是卡点清单，不是写回草稿。

## 与多智能体审计的关系

- 多智能体审计：多个审计角色从科研、工程、复用和论文评审角度独立施压。
- walkthrough：一个 agent 按线性顺序扮演新读者，不对抗任何立场，只记录自己的理解与卡点。

二者可串联使用：walkthrough 发现观点性争议点时，进入 `report-audit` 多智能体审计。

## 输入契约

启动前，audit 主 agent 应确认以下输入：

- 报告工作区路径。
- 读者画像。默认：认真、愿意学、不预设术语；不熟悉所在领域的主流工具；阅读语言为中文。
- 走查路径。默认：取自 `reading-path-map.md` 中「新成员 精读」一行，即从 00 到最末章按章节顺序。
- 关注章节。默认全量。当用户只关心单章或单节的可读性时，允许缩窄范围。
- 终止条件。默认读完全部章节或累计卡点超过 20 条。

这些输入应写进 walkthrough 的开场备忘里，方便 audit 主 agent 判断协议是否被遵守。

## 走查步骤

walkthrough agent 按以下顺序执行：

1. 打开报告工作区，读 `brief.yaml` 摘录报告 `title`、`report_type`、`audience`。
2. 进入 `00-executive-summary`，读完后记录：
   - 本节核心结论是什么。
   - 我现在能复述的新判断是什么。
   - 我卡在哪里（若无则明确写「无」）。
3. 按走查路径顺序读其余章节，每节结束前重复「三问」记录。
4. 每 3 章做一次「心智盘点」：
   - 我能否画出系统总图。
   - 我能否列出主要模块。
   - 我能否复述主要 tradeoff。
5. 走完全部章节后，产出卡点清单。

走查过程中不得：

- 回头修改已经记录的三问；
- 查阅报告之外的资料（外部背景知识允许，但必须显式标注来源）；
- 编辑报告源文件。

## 三问模板

每一节结束前，按下述最小结构记录：

- 位置：章节号与小节名。
- 核心结论：用一句复述。
- 新判断：一到三条。
- 卡点：位置 + 类型 + 说明。若无卡点，写「无」。

卡点类型取自以下集合，可组合：

- `术语未定义`：出现未定义或定义位置晚于使用的术语。
- `证据层混淆`：把 design intent 当作 repo-observed fact，或反之。
- `跳跃过大`：读者在本节前未获得理解本节所需的前置判断。
- `依赖未声明`：命令、接口、凭证的前置未在更早章节说明。
- `图不能解释结论`：图存在但无法单独支撑章节的主结论。
- `结论无出处`：判断未指向证据来源。
- `心智碎片化`：连续 2 节后仍无法把新信息组织进同一心智模型。

其他类型按需扩展，但需在输出里说明。

## 输出形状

最终输出一份 Markdown 或纯文本清单，列如下三列：

- 位置（`sections/04-system-architecture.tex:42` 或更精确的定位）。
- 卡点类型（取自集合或扩展）。
- 建议修复（一句话，不写完整改稿）。

输出末尾追加三段：

- 心智盘点结果：画图能力、模块列举能力、tradeoff 复述能力。
- 是否建议进入 `report-audit` 多智能体审计，给出候选命题与目标章节。
- 是否建议触发 `report-update`，给出待写回优先级。

## 写回规则

walkthrough 默认不改报告。卡点清单作为 `report-update` 的输入。

当 audit 主 agent 想要把 walkthrough 结论直接写回时，必须先与用户确认：

- 是否接受卡点清单中的某一条作为下一次写回的目标；
- 目标章节与改写边界；
- 是否联动相邻章节做一致性扫面。

## 与 checklist 的协作

walkthrough 产出的卡点可反向映射到：

- `mental-model-checklist.md` 的 8 条验收问题。
- `implementation-readiness-checklist.md` 的 7 类验收项。

映射关系帮助用户判断「这些卡点到底挂在心智缺口还是实现缺口上」，从而决定修复优先级。

## 质量自检

协议结束前检查：

- 三问是否每节都完成；
- 卡点清单中每条是否都标了类型；
- 心智盘点是否每 3 章都触发过；
- 输出是否保留了完整走查痕迹，而不是只给一份压缩结论。
