# 正式中文报告风格（金字塔 + 图解版）

本文件定义 `report` 家族的默认表达风格。**核心原则：金字塔原理 + 图解优先 + 信息密度最大化。**

目标读者是计算机专业人士。不要 ELI5 类比，不要大段铺垫，不要重复解释。能用一张图说清的，不要用三段文字。

---

## 一、金字塔原理（第一原则）

### 1.1 结论先行

每一章、每一节、每一段都必须先给结论，再给支撑。

**正确结构：**
```
[结论/判断] → [MECE 分解的关键支撑点] → [细节展开] → [技术深潜]
```

**禁止结构：**
```
[背景铺垫] → [逐步引导] → [最后给结论]
```

### 1.2 MECE 分解

支撑结论的要点必须**MECE**（Mutually Exclusive, Collectively Exhaustive）：

- **相互独立**：各要点不重叠
- **完全穷尽**：合起来覆盖全部

**示例（MECE）：**
```markdown
## 核心组件

系统拆为三层：接入层、执行层、持久层。

| 层级 | 职责边界 | 关键技术 |
|------|----------|----------|
| 接入层 | 协议转换与请求校验 | REST / WebSocket |
| 执行层 | 业务逻辑编排 | Temporal Workflow |
| 持久层 | 状态存储与查询 | Prisma + PostgreSQL |
```

**反例（非 MECE）：**
```markdown
系统包含以下模块：用户管理、订单处理、数据存储、日志记录、错误处理。
```
（错误处理横跨所有模块，不是独立层级）

### 1.3 逐层展开

每层只展开当前层次需要的细节，下一层再深入。

```markdown
## 4. 系统架构

### 4.1 总览（层 1）
[一句话结论 + 一张总览图]

### 4.2 接入层细节（层 2）
[接入层的 MECE 分解 + 层内架构图]

### 4.2.1 认证模块（层 3）
[模块架构图 + 核心接口 + 关键决策]

### 4.2.2 路由模块（层 3）
[模块架构图 + 核心接口 + 关键决策]
```

## 二、图解优先（第二原则）

### 2.1 图文替换规则

| 文字描述长度 | 替换为 |
|--------------|--------|
| 超过 3 句的流程描述 | 一张流程图 |
| 超过 2 段的模块关系 | 一张架构图 |
| 超过 1 页的交互说明 | 一张时序图 |
| 多方案对比 | 一张对比表或矩阵图 |

**原则**：如果读者需要在大脑中"重建"结构才能理解，就必须画图。

### 2.2 图表策略（decode 兼容）

沿用 decode skill 的图表策略：

1. **先给总览图**：只展示参与者、层级聚合、外部系统
2. **再补细节图**：按层级或职责簇分别展开
3. **超过 5 个内部节点或 2 个外部依赖**：拆成独立图
4. **无内部结构的层级**：省略细节图

### 2.3 Mermaid 统一规范

**默认规范：**
```markdown
flowchart TB
    subgraph web["Web 层"]
        page["页面"]
        hook["Hook"]
    end
    
    subgraph api["API 层"]
        ctrl["控制器"]
        svc["服务"]
    end
    
    page -->|"调用"| hook
    hook -->|"HTTP"| ctrl
    ctrl -->|"委托"| svc
```

**硬性约束：**
- 默认 `flowchart TB`
- `subgraph id["中文标题"]`
- 节点 ID 用 ASCII，显示文本用中文
- 边标签用简短动词（调用、返回、触发、写入）
- Mermaid 主要用于 brainstorming 草图；数据流图、状态图和扇出图只有在当前工作区存在稳定 Mermaid-to-PDF 渲染链路、或用户明确接受该形式时，才可以作为正式稿。否则转为 TikZ / PGFPlots 或表格。

### 2.4 每章配图规则

| 章节类型 | 必须包含 |
|----------|----------|
| 系统架构 | 总览图 + 各层细节图 |
| 模块设计 | 模块架构图 + 核心接口表 |
| 核心流程 | TikZ 流程图或 TikZ 时序图 |
| 接口契约 | Mermaid source 数据流图或 TikZ 数据流图 + 字段定义表 |
| 实现走查 | TikZ 关键链路时序图 |

**禁止**：一章超过 5 页没有任何图或表。

## 三、模块级细节（第三原则）

### 3.1 模块文档标准结构

每个模块必须提供：

```markdown
## [模块名] 模块

### 1. 治理思想
**一句话定义这个模块的本质。**

### 2. 组件架构（MECE）
[模块内部架构图]

| 组件 | 职责边界 | 关键技术 |
|------|----------|----------|
| ... | ... | ... |

### 3. 接口契约
[核心接口定义]

### 4. 数据结构
[关键类型 / 表结构]

### 5. 设计决策
[为什么这样设计，主要权衡]

### 6. 交互流程
[时序图：展示与外部模块的交互]
```

### 3.2 Sequence Chart 使用场景

**必须使用时序图：**
- 跨模块调用链路（>2 个参与者）
- 异步消息交互（信号、事件、回调）
- 状态转换依赖时序（状态机）
- 重试/补偿/幂等逻辑

**时序图规范（正式报告默认 TikZ）：**
```latex
\begin{figure}[H]
\centering
\input{figures/<sequence-slug>.tex}
\caption{[对象]，突出[读者应注意的结论]。}
\label{fig:<sequence-slug>}
\end{figure}
```

**补充约束：**
- 用户提供的 PNG、截图或 Mermaid 时序图，默认视为 TikZ 重绘参考
- 主链路时序图控制在 4--8 个参与者
- 超时、重试、补偿、异常分支应拆成独立图或独立表
- 图前说明这张图回答什么问题，图后说明主链路中的阻塞点或关键裁决点

## 四、高密度写作规范

### 4.1 段落纪律

- **每段只做一件事**：定义 / 动机 / 方法 / 证据 / 解释 / 风险
- **段落长度**：3-6 句，超过 8 句必须拆分
- **每段必须有判断**：禁止纯描述性段落

### 4.2 句子纪律

- **一句一个判断**：不要无限加插入语
- **禁止对照否定句**：`不是...而是...`、`真正重要的不是...`
- **禁止空泛强调**：`本质上`、`从某种意义上说`
- **禁止行内串列**：不要用`第一...第二...第三...`塞在同一段

### 4.3 分点纪律

**显式分点必须使用 LaTeX 列表：**
```latex
\begin{itemize}
    \item 要点一
    \item 要点二
\end{itemize}
```

**禁止在 `.tex` 正文中写：**
```latex
- 要点一
* 要点二
1. 要点三
```

### 4.4 信息密度自检

每段写完后问自己：

1. **删掉前半句，信息是否几乎不变？** → 前半句是废话
2. **读者需要这句话才能理解下一句吗？** → 如果不是，考虑删除
3. **这个概念在其他地方解释过吗？** → 交叉引用，不要重复
4. **能用图/表替代这段文字吗？** → 优先图解

## 五、章节结构模板

### 5.1 系统架构章节

```markdown
## X. 系统架构

### X.1 总览
[一句话结论：系统拆为 N 层，解决...问题]

[总览图]

图 X. 系统层级总览，突出[关键边界/依赖关系]。

### X.2 [层级 A]
[一句话职责]

[层级 A 架构图]

| 组件 | 职责 | 关键技术 |
|------|------|----------|
| ... | ... | ... |

### X.3 [层级 B]
...
```

### 5.2 模块设计章节

```markdown
## X. [模块名] 模块设计

### X.1 治理思想
[一句话本质定义]

### X.2 组件架构
[模块架构图]

### X.3 核心接口
[接口定义表或代码片段]

### X.4 交互流程
[时序图]
```

### 5.3 接口契约章节

```markdown
## X. 接口契约

### X.1 [接口名]

**职责**：[一句话]

**输入/输出**：[类型定义]

**数据流**：[流程图]

**错误码**：[表格]
```

## 六、禁止清单

### 6.1 禁用表达

- `不是……而是……`
- `并不是……而是……`
- `真正重要的不是……`
- `真正决定……的不是……`
- `真正关注的不是……`
- `真正的难点不在……`
- `换句话说`（后面没有新增信息）
- `从某种意义上说`
- `本质上`（后面没有新增定义、机制或边界）
- `众所周知`
- `显而易见`

### 6.2 禁用结构

- **纯背景铺垫章节**：不要整章只讲"为什么做这个项目"
- **重复解释**：已经在术语表定义的概念，正文里不要再解释
- **无图模块介绍**：介绍模块必须有架构图
- **无结论细节**：给出细节前必须先说明这个细节支撑什么结论

### 6.3 禁用图表

- **装饰性图表**：不能帮助理解结论的图
- **过度拥挤的图**：一张图超过 9 个节点或混合多个主题
- **无图注的图**：每张图必须有结论性图注

## 七、自检清单

提交前必须检查：

- [ ] 每章开头是否有明确结论句？
- [ ] 支撑结论的要点是否 MECE？
- [ ] 每章是否有至少一张图或表？
- [ ] 模块介绍是否有架构图？
- [ ] 跨模块交互是否有时序图？
- [ ] 是否有禁用表达？
- [ ] 是否有纯描述性段落？
- [ ] 是否有与其他章节重复的内容？
- [ ] 图解是否替代了冗长文字？
- [ ] 段落是否都控制在 3-6 句？

## 八、与 decode skill 的对齐

本风格与 `decode` skill 保持兼容：

| decode 输出 | report 章节 |
|-------------|-------------|
| 治理思想 | 模块设计 → 治理思想 |
| 组件架构（MECE） | 系统架构 / 模块设计 → 组件架构 |
| 接口契约 | 接口契约章节 |
| 数据结构 | 数据模型章节 |
| 设计决策 | 设计决策章节 |
| 交互流程 | 时序图章节 |

**原则**：decode 的终端分析结果可以直接作为 report 对应章节的输入，结构和术语保持一致。
