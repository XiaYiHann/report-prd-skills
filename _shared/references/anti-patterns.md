# 低信息密度反面教材（扩展版）

本文件收集 `docs/report/**` 中已经出现过的低信息密度表达，以及**图解缺失、金字塔结构破坏、模块细节不足**等新型反例。重点不是批评旧稿，而是把常见坏味道显式写出来，避免以后反复出现。

默认原则：
- **结论先行**，禁止大段铺垫
- **图解优先**，能用图说的不用字
- **MECE 分解**，支撑要点相互独立、完全穷尽
- **直接陈述真实信息**，不要先搭一个否定框架再把答案拖到后半句
- **强调句必须带来新增信息**，不能只制造语气
- **若一句话删掉前半句后信息几乎不变，这句话通常就太稀了**

## 反面模式 1：对照否定句

### 例 1

- 来源：`path/to/project/docs/report/sections/01-executive-summary.md:3`
- 原句：`目标不是直接替你写论文，而是先把这个科研项目讲清楚，再把"怎么上手实验"讲明白。`
- 问题：前半句只是在排除一个并不重要的误解，真正有效的信息只有"这份报告用于讲清项目与实验上手路径"。
- 改写原则：直接写报告职责，不要先虚构一个错误目标再反转。
- **改写**：`本报告阐明科研项目逻辑与实验上手路径。`

### 例 2

- 来源：`path/to/project/docs/report/sections/02-problem-and-motivation.md:5`
- 原句：`真正决定 MoE 是否工作良好的不是"专家数量"，而是"router 把 token 送给谁"。`
- 问题：句子把重点放在修辞对照上，而不是把判断拆清楚。
- 改写原则：直接指出主要决定因素及其作用机制。
- **改写**：`路由分配质量比专家数量更影响协同收益。高质量路由需满足负载均衡与专家专业化两个约束。`

## 反面模式 2：空泛强调句

### 例 3

- 来源：`path/to/project/docs/report/sections/02-problem-and-motivation.md:7`
- 原句：`这个项目真正关注的不是"如何再加一个专家"，而是"如何让已经存在的专家组合得更合理"。`
- 问题：读者看到的是"真正关注"这种强调词，而不是清晰的研究对象、变量和约束。
- 改写原则：直接写研究目标。
- **改写**：`本项目研究固定专家集合下的组合优化策略，目标是提升协同收益。`

## 反面模式 3：大段背景铺垫

### 例 4

- 类型：整章只讲"为什么做这个项目"
- 原结构：
  ```markdown
  ## 背景与上下文
  
  在加密货币交易领域，自动执行一直存在两个极端...
  一端是脚本拼接...
  另一端是机构级交易平台...
  Ares 面对的环境约束有...
  1. 多租户隔离...
  2. 信号与执行分离...
  3. 安全边界...
  4. 渐进演化...
  ```
- 问题：2 页背景后才触及项目本身。读者已经知道"为什么"才来看报告的。
- 改写原则：**SCQA 压缩到 1 页以内**。先给结论（Ares 是什么），再用一张痛点-回答对照图替代大段文字。
- **改写**：
  ```markdown
  ## 背景与上下文
  
  Ares 是一套面向多租户的 agent-native 交易执行平台，解决"信号到执行的可靠转换"。
  
  [痛点-回答对照图]
  
  | 痛点 | Ares 回答 |
  |------|-----------|
  | 信号与执行耦合 | Agent 模型解耦 |
  | 缺乏可审计性 | 事件优先持久化链 |
  | 风控事后补丁 | 执行主路径 first-class 风控 |
  | 密钥管理随意 | Envelope encryption |
  ```

## 反面模式 4：该有图没图

### 例 5

- 类型：模块介绍没有架构图
- 原文：
  ```markdown
  ### Signal Agent 模块
  
  Signal Agent 负责生成交易信号。它接收市场数据输入，运行策略算法，
  输出方向性信号。信号包含标的、方向、建议仓位等字段。
  策略算法支持多种类型，包括趋势跟踪、均值回归等。
  信号生成后通过事件总线分发到订阅者...
  ```
- 问题：300 字描述模块结构，读者必须在大脑中"重建"架构。
- 改写原则：**模块介绍必须先给架构图，再用表格列组件职责，最后才文字补充**。
- **改写**：
  ```markdown
  ### Signal Agent 模块
  
  **治理思想**：将市场数据转换为标准化交易信号的可插拔策略引擎。
  
  [模块架构图]
  
  | 组件 | 职责 | 关键技术 |
  |------|------|----------|
  | Data Ingestor | 接收并标准化市场数据 | WebSocket / REST 适配器 |
  | Strategy Engine | 运行策略算法 | 插件化策略加载器 |
  | Signal Emitter | 生成并分发信号事件 | 事件总线 |
  ```

### 例 6

- 类型：流程描述没有流程图
- 原文：
  ```markdown
  信号从生成到执行的流程如下：首先，Signal Agent 生成信号并写入 
  signal_event 表；然后，Signal Delivery 服务读取信号，根据订阅关系
  找到目标 Trader Agent；接着，风控服务评估该信号是否允许执行；
  如果通过，则创建 execution_task 并交由 Worker 执行；Worker 调用
  交易所 API 下单，并将结果写回 exchange_order 表...
  ```
- 问题：10 句话描述一个流程，读者需要逐句跟踪状态转移。
- 改写原则：**超过 3 句的流程描述必须替换为流程图或时序图**。
- **改写**：
  ```markdown
  [时序图：Signal Agent → Delivery → Risk → Worker → Exchange]
  
  图 X. 信号执行链路，突出风控决策点在主路径中的位置。
  ```

## 反面模式 5：重复解释

### 例 7

- 类型：术语表已定义，正文中再解释
- 原文：
  ```markdown
  ### Signal Agent
  
  Signal Agent（信号代理）是一种生成交易信号的自动化组件。
  在 Ares 系统中，Signal Agent 负责...（下接 200 字解释）
  ```
- 问题：术语表已经定义了 Signal Agent，正文再解释一遍。
- 改写原则：**术语表定义过的概念，正文直接引用，不再重复解释**。
- **改写**：
  ```markdown
  ### Signal Agent
  
  Signal Agent 生成标准化交易信号（详见术语表 §3.1）。
  
  [架构图]
  ```

## 反面模式 6：无结论细节

### 例 8

- 类型：直接罗列细节，不说这些细节支撑什么结论
- 原文：
  ```markdown
  ### 接口字段
  
  - `id`: string, 必填
  - `name`: string, 必填
  - `status`: enum('active', 'inactive'), 必填
  - `environment`: enum('sandbox', 'production'), 必填
  - `exchangeAccountId`: string, 可选
  - `createdAt`: datetime
  - `updatedAt`: datetime
  ```
- 问题：罗列了字段，但没有说明"为什么这些字段这样设计"。
- 改写原则：**细节前必须有结论，细节必须支撑结论**。
- **改写**：
  ```markdown
  ### 接口字段
  
  **设计原则**：最小必要 + 环境隔离 + 延迟绑定交易所账户。
  
  | 字段 | 必填 | 设计理由 |
  |------|------|----------|
  | id, name, status | 是 | 最小身份标识 |
  | environment | 是 | 强制环境隔离 |
  | exchangeAccountId | 否 | 允许先创建 Agent 后绑定账户 |
  ```

## 反面模式 7：装饰性图表

### 例 9

- 类型：为了"图文并茂"而插入无关图表
- 原文：在"编码规范"章节插入一张"Git 分支模型图"。
- 问题：编码规范的核心是命名规则和目录结构，Git 分支模型与当前主题无关。
- 改写原则：**每张图必须回答当前章节的一个核心问题**。不能帮助理解结论的图必须删除。

---

## 反面模式 8：过度拥挤的图

### 例 10

- 类型：一张图塞入所有信息
- 原文：一张架构图包含 15+ 节点，同时展示 Web 层、API 层、Worker 层、数据层、外部系统、部署拓扑。
- 问题：读者无法一眼识别关键边界。
- 改写原则：**超过 9 个节点或混合多个主题的图必须拆分**。
- **改写**：
  ```markdown
  [图 1：系统总览，4 个层级节点]
  [图 2：API 层细节，5 个模块节点]
  [图 3：Worker 层细节，4 个活动节点]
  ```

## 反面模式 9：纯描述性段落

### 例 11

- 类型：段落只有描述，没有判断
- 原文：
  ```markdown
  系统采用前后端分离架构。前端使用 React 框架，后端使用 NestJS 框架。
  数据库使用 PostgreSQL，缓存使用 Redis。系统部署在 Docker 容器中。
  ```
- 问题：罗列技术栈，没有说明"为什么选这些技术"。
- 改写原则：**每段必须有判断：为什么选 A 而不是 B，A 解决了什么问题**。
- **改写**：
  ```markdown
  系统采用前后端分离架构。前端 React 提供组件化 UI 复用；后端 NestJS
  的模块化设计匹配领域驱动拆分。PostgreSQL 作为主库存储关系型业务数据，
  Redis 缓存高频读路径以降低数据库压力。
  ```

## 反面模式 10：行内串列枚举（升级版）

### 例 12

- 类型：在 `.tex` 中用伪列表
- 原文：
  ```latex
  系统核心组件包括：
  - Signal Agent，负责信号生成
  * Trader Agent，负责执行上下文
  1. Risk Service，负责风控决策
  ```
- 问题：看起来像列表，但没有进入 LaTeX 列表结构。
- 改写原则：**显式分点必须使用 `itemize` / `enumerate` + `\item`**。
- **改写**：
  ```latex
  系统核心组件包括：
  \begin{itemize}
      \item Signal Agent：信号生成
      \item Trader Agent：执行上下文
      \item Risk Service：风控决策
  \end{itemize}
  ```

## 建议替代写法汇总

遇到上面的句型时，优先采用以下模式：

1. **直接给结论**，再用下一句解释原因、机制或边界
2. **用图/表替代冗长文字**
3. **模块介绍 = 架构图 + 组件表 + 一句话补充**
4. **流程描述 = 时序图/流程图 + 关键节点说明**
5. **细节前先说结论**，细节必须支撑结论
6. **术语表已定义的概念，正文直接引用**

## 默认禁用表达（完整清单）

最终成稿默认禁止或强烈限制以下表达：

| 类型 | 表达 | 原因 |
|------|------|------|
| 对照否定 | `不是……而是……` | 前半句废话 |
| 对照否定 | `并不是……而是……` | 前半句废话 |
| 空泛强调 | `真正重要的不是……` | 只制造语气 |
| 空泛强调 | `真正决定……的不是……` | 只制造语气 |
| 空泛强调 | `真正关注的不是……` | 只制造语气 |
| 空泛强调 | `真正的难点不在……` | 只制造语气 |
| 空泛转换 | `换句话说`（无新增信息） | 重复表达 |
| 空泛限定 | `从某种意义上说` | 模糊边界 |
| 空泛本质 | `本质上`（无新增定义） | 伪深刻 |
| 虚假共识 | `众所周知` | 推卸解释责任 |
| 虚假共识 | `显而易见` | 推卸解释责任 |
| 行内串列 | 同一段中 `第一……第二……第三……` | 阅读负担高 |
| 行内串列 | 同一段中 `其一……其二……` | 阅读负担高 |
| 伪列表 | `.tex` 中写 `- `、`* `、`1. ` | 未进入列表结构 |
